1 Description

[hide]

Builds a seamless mosaic from a set of plate-solved input tiles. The pipeline runs in two phases.

Phase 1 - Registration. Every input tile is reprojected onto a common astrometric frame using MosaicByCoordinates. The script pre-fills the input list, output directory, suffix, and overwrite flag, then launches MosaicByCoordinates so its dialog opens with everything in place. Confirm the projection and mosaic dimensions and click OK. MosaicByCoordinates writes one <name><suffix>.xisf per tile into the chosen output directory.

Phase 2A - GradientMergeMosaic. On the Process path, the registered tiles are fed to GradientMergeMosaic, which merges them in the gradient domain to eliminate seams. This is the recommended path - the process runs natively on the V8 runtime and works today.

Phase 2B - Mosaic scripts. On the Scripts path, the registered tiles are opened as windows and the script tries to launch TrimMosaicTile (optional) and PhotometricMosaic via Process.executeScriptGlobal. These scripts are still on the SpiderMonkey runtime as of PixInsight 1.9.4 and will fail to load; the pipeline will report the error and stop. This mode is wired up so it will work transparently as soon as those scripts are ported to V8 - no changes to the pipeline needed.

Both modes require MosaicByCoordinates in Phase 1; it is the only tile-registration step the pipeline performs.

2 Usage

[hide]

Open the script from Scripts > EGo > Mosaic Pipeline. Each input tile must already have a valid astrometric solution (ImageSolver, or solver-equipped acquisition software). Tiles that are not plate-solved will be rejected by MosaicByCoordinates in Phase 1.

Input tiles

File list of source tiles. Use Add Files... to pick FITS/XISF images; Remove drops selected entries; Clear empties the list. At least two tiles are required.

Output directory

Where MosaicByCoordinates writes the registered tiles. Must exist. The same directory is read back in Phase 2 to locate the merge inputs, so any external change to file names between phases will break the pipeline.

Registered suffix

Suffix appended to each registered tile's basename. Defaults to _ra to match MosaicByCoordinates' own default.

Overwrite existing

Forwarded to MosaicByCoordinates; if off and a registered file already exists, MBC will abort.

GradientMergeMosaic mode

Runs the GMM process on the registered tiles. Average overlap maps to GMM's Average combination type (better SNR on overlapping regions, order-independent); unchecking it selects Overlay. Shrink removes border pixels affected by interpolation aliasing - the default of 1 is right for most tiles. Feather controls the seam transition radius in pixels.

Scripts > Mosaic mode

Opens the registered tiles as windows and attempts to launch TrimMosaicTile (if checked) and PhotometricMosaic (if checked). Currently warns up front - the underlying scripts do not yet run on the V8 runtime.

3 Pipeline stages

[hide]

3.1 Phase 1: registration

Pre-fills engine_* parameters (the keys PersistentObject.LoadParameters consumes for MosaicByCoordinatesEngine, whose prefix is "engine"), then calls Process.executeScriptGlobal("MosaicByCoordinates/MosaicByCoordinates"). After MBC returns, the script computes the expected output paths (<outputDir>/<basename><suffix>.xisf) and aborts if any are missing.

3.2 Phase 2A: GradientMergeMosaic

Builds targetFrames as [[true, path], ...] and runs the GMM process with the user-selected combination type, shrink, and feather. The merged mosaic is left as an open window for further linear processing.

3.3 Phase 2B: Mosaic scripts (deferred)

Opens each registered tile as a window. Attempts to launch JohnMurphy/PhotometricMosaic/TrimMosaicTile and JohnMurphy/PhotometricMosaic/PhotometricMosaic via Process.executeScriptGlobal, wrapped in try/catch. On failure (the expected outcome today) it logs a clear note that the script is still on SpiderMonkey and continues with the next one if any.

4 Notes

[hide]

4.1 Plate solutions

Every input tile must carry a usable astrometric solution. Phase 1 fails fast if a tile lacks one - solve missing tiles with ImageSolver (Scripts > Image Analysis) and re-run.

4.2 Linear vs nonlinear input

Both merge backends accept linear or nonlinear data and preserve linearity where applicable, but mixing linear and nonlinear tiles in the same run will produce visible brightness mismatches that no merge can hide. Keep tiles in the same processing state.

4.3 Parameter-passing fragility

Phase 1 relies on MosaicByCoordinatesEngine keeping its "engine" PersistentObject prefix. If a future MosaicByCoordinates version changes that prefix, the pre-fill will silently no-op and the MBC dialog will open empty. The pipeline will still run if the user populates MBC manually, but the output-path check at the end of Phase 1 will fail unless the chosen suffix and output dir match what was passed to this script.
